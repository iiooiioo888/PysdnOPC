本仓库采用按领域划分的自定义异常类与标准 RuntimeError/ValueError 混用的错误处理模式，未建立统一的错误基类或全局错误码枚举。各层模块自行定义语义明确的异常类型，并在调用链中向上抛出，由上层（CLI、UI 服务）统一捕获并转换为结构化响应。

## 1. 已定义的领域异常
- opc/core/org_config.py: RunnableOrgConfigError(ValueError) - 组织配置校验失败
- opc/layer2_organization/collaboration_service.py: CommunicationDeliveryError(RuntimeError) - 协作消息投递失败
- opc/layer2_organization/talent_market.py: HireError(Exception), RoleAlreadyHiredError(HireError) - 招聘流程错误
- opc/layer3_agent/preflight.py: ExternalAgentPreflightError(RuntimeError) - 外部 Agent 预检失败
- opc/layer4_tools/file_ops.py: PatchApplyError(RuntimeError) - 补丁应用失败
- opc/layer5_memory/skill_importer.py: SkillImportError(RuntimeError) - 技能导入失败
- opc/llm/retry.py: LLMRetryError(RuntimeError) - LLM 重试耗尽
- opc/plugins/office_ui/services/models.py: ServiceError(Exception) - Office UI 服务层通用错误（带 code/message/context）
- opc/plugins/office_ui/ws_handler.py: ProjectScopeError(ValueError) - WS 项目范围校验错误

## 2. 错误传播与呈现
- CLI 层 (opc/cli/app.py): 通过 except ServiceError as exc: 捕获服务层错误，将 code/message/context 字段输出为结构化 JSON 响应；对 ValueError 等做兜底处理。
- Channel 层 (opc/channels/*.py): 大量直接使用 raise RuntimeError(...) 表达渠道特定错误（如飞书发送失败、Slack 模式不支持），未封装为自定义异常，属于快速失败风格。
- LLM 重试 (opc/llm/retry.py): 在多次重试后抛出 LLMRetryError，供上层区分网络/配额类可恢复错误与业务错误。

## 3. 缺失与不一致
- 无统一错误基类：现有异常分别继承 Exception、RuntimeError、ValueError，无法通过单一基类统一捕获。
- 无全局错误码表：仅 ServiceError 使用字符串 code，其他异常依赖 Python 异常类型区分。
- 未使用 panic/recover：Python 生态下未见 try/finally 资源清理之外的恢复逻辑。
- 中间件式错误处理缺失：除 CLI 和 Office UI 的局部 try/except 外，核心运行时链路缺乏横切错误拦截器。

## 4. 开发者约定建议
1. 新增领域错误时优先定义独立异常类（参考 HireError → RoleAlreadyHiredError 的层次结构）。
2. 跨进程/跨协议边界（CLI↔Office UI、外部 Agent↔公司运行时）应使用带 code/message/context 的结构化错误对象。
3. 底层 I/O 错误优先抛出自定义异常而非裸 RuntimeError，以便上层区分可重试与不可恢复场景。
4. 避免在深层工具函数中吞掉异常，让错误沿调用栈向 CLI/UI 层冒泡集中处理。
